Anti-Patterns

What to Avoid

Patterns that feel right but cause the exact problems we're trying to solve.

Invisible State

Any pattern where tensor device/dtype is inferred rather than explicit. This includes thread-locals, module-level globals, and "smart" defaults that guess based on other tensors. If you can't see where the device comes from by reading the function signature, it's invisible state.

Distributed Device Inference

N independent decisions that must all agree: torch.cuda.is_available() checks scattered throughout the codebase. Each one is a potential disagreement. Replace with single source of truth.

Ad-hoc Type Coercion on Checkpoint Restore

Scattered int(), float(), and .get() calls for checkpoint fields. Each is a potential silent corruption. Schema versioning with explicit validation is the fix.

Premature Convenience Wrappers

Adding ctx.zeros(), ctx.tensor(), ctx.randn(), ctx.ones() before usage patterns emerge. Each wrapper is API surface that must be maintained. Start with ctx.device and ctx.dtype — add helpers only when 3+ call sites repeat the same pattern.

Root Cause

Why These Bugs Existed

Architectural Weakness

No Centralized Device/Dtype Context

Each component independently infers tensor placement, creating N independent decisions that must all agree. When components are written by different people at different times, they make different assumptions. Some check torch.cuda.is_available(), some infer from input tensors, some have hardcoded defaults.

The result: 18+ bugs across 3 hardening rounds, all variations of "tensor ended up on wrong device" or "checkpoint field had wrong type after restore."
