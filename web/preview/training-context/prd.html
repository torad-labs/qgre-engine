Core Type

TrainingContext Dataclass

A simple dataclass with device, dtype, step, and checkpoint_schema_version. Created once by QGRETrainer, passed explicitly to all tensor-creating components. No ambient state, no thread-locals, no magic.

@dataclass
class TrainingContext:
    device: torch.device
    dtype: torch.dtype = torch.float32
    step: int = 0
    checkpoint_schema_version: int = 1

    @classmethod
    def from_config(cls, config: QGREConfig, device: str = "cuda") -> "TrainingContext":
        return cls(
            device=torch.device(device),
            dtype=getattr(torch, config.dtype, torch.float32),
            step=0,
            checkpoint_schema_version=CHECKPOINT_SCHEMA_VERSION,
        )

    def to_dict(self) -> dict:
        # For checkpoint serialization
        return {"device": str(self.device), "dtype": str(self.dtype), ...}

Build Sequence

Implementation Order

  1. Foundation: qgre/types.py

    Add TrainingContext dataclass + CHECKPOINT_SCHEMA_VERSION constant. No dependencies.

  2. Core: qgre/trainer.py

    Create self.ctx in __init__, update ctx.step each training step, pass to components.

    Depends on: Phase 1

  3. Consumers (parallel)

    Update advantages.py, critic.py, spans.py, weight_load.py to receive and use ctx.

    Depends on: Phase 2

  4. Persistence: qgre/checkpoint.py

    Serialize ctx.to_dict() in checkpoint, validate schema_version on restore.

    Depends on: Phases 1-2

  5. Cleanup

    Remove all torch.cuda.is_available() device inference scattered throughout codebase.

    Depends on: Phases 3-4

Helper Methods (uncertainty: 0.5)

Should we add ctx.zeros(shape), ctx.tensor(data) convenience wrappers? Start minimal with ctx.device only. Add helpers when 3+ call sites repeat the same pattern.

Schema Migration (uncertainty: 0.4)

What happens when schema_version mismatches on checkpoint restore? Options: raise error, auto-migrate, warn and continue. Recommend: raise clear error with migration instructions.

Step Mutability (uncertainty: 0.3)

Mutate ctx.step in-place or create new ctx per step? Either works. Recommend: mutate in-place for simplicity.

Emergence

What Surprised Us

The strength of the ambient context pull

Thread-local "elegant" patterns felt obviously right until the evidence from 43 hardening bugs made clear: invisible state is the enemy. The explicit, slightly-verbose parameter passing that felt like "boilerplate" is actually the architectural choice that prevents entire bug classes.
