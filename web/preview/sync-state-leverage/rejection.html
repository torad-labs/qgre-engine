Rejection Framework

What NOT to Do

Anti-patterns, overturned assumptions, and boundaries. These approaches were considered and rejected during crystallization.

Anti-Patterns

Approaches That Would Fail

Add More Assertions (v4)

The temptation: keep flags scattered but add consistency assertions everywhere. "Defense in depth."

Why rejected: 10 rounds of hardening already added assertions. They caught bugs but didn't prevent the bug class. Detection is not prevention. Structural fix > runtime checks. Assertions are O(n) in bug count; structural impossibility is O(1).

Bundle in TrainingContext (v2)

The temptation: TrainingContext is already passed everywhere. Just add SyncState as a field.

Why rejected: TrainingContext is a config dataclass — immutable, passed per-call, belongs to the outer loop. SyncState is runtime state — mutable, needs to live across calls, belongs to the inner loop. Bundling conflates config with state. WeightBus and WeightLoader don't receive TrainingContext at construction; they get it per sync call. Wrong level of abstraction.

Singleton / Global State (training pull)

The temptation: create a module-level SYNC_STATE singleton that anyone can import.

Why rejected: Singletons are implicit dependencies. Tests become impossible to isolate. Multiple trainers (future possibility) would share state incorrectly. DI is explicit and testable. The existing test_sync_state.py works because SyncState is instantiated per test.

SyncState Owns the Operations (v6 - partial)

The temptation: move apply_lora_dropout logic INTO SyncState. Make it state.apply_dropout(model, rate).

Why rejected: SyncState should be generic state management. apply_lora_dropout has LoRA-specific logic (finding lora_A parameters, Bernoulli masking, weight cloning). Coupling domain logic to state management violates single responsibility. The compromise: SyncState provides the context manager, apply_lora_dropout does the work inside it.

Event Sourcing (v8)

The temptation: every state change is an event. Reconstruct state by replaying. Audit trail for debugging.

Why rejected: Over-engineering. This is a single-machine training loop, not a distributed system. The state space is 5 booleans. The "audit trail" is logging. Event sourcing adds complexity without solving the actual problem (scattered ownership). YAGNI.

Protocol / Interface Abstraction (v7)

The temptation: define class SyncStateProtocol(Protocol) so we can inject mocks.

Why rejected: Premature abstraction. SyncState is already testable — test_sync_state.py proves it. The class is concrete, has no I/O, and can be instantiated directly in tests. Adding a Protocol adds indirection without benefit. If we need mocking later, we can add it then.

Overturned

Assumptions That Were Wrong

Assumption What Evidence Showed Impact
TrainingContext is the natural home for runtime state TrainingContext is config (immutable, passed per-call). SyncState is runtime (mutable, lives across calls). Different lifecycles. SyncState is a separate object with DI
State objects should only hold state, not operations The restore() pattern (return a cleanup function) is forgotten. Context managers guarantee cleanup via language construct. SyncState provides dropout_context()
Module-level locks are fine for synchronization _dropout_lock in lora_dropout.py is separate from _sync_lock in weight_bus.py. Two locks protecting related state = coordination nightmare. Single lock inside SyncState._lock
More checks = more safety 10 rounds of hardening added assertions. They caught bugs at runtime but didn't prevent the bug class. Detection != prevention. Structural fix (impossible states) over runtime checks
Boundaries

Hard Limits

Do NOT add SyncState to TrainingContext. They have different lifecycles. Config is immutable and passed per-call. State is mutable and persists across calls.
Do NOT make SyncState a singleton. Use dependency injection. Singletons are implicit dependencies that break testing and future multi-trainer scenarios.
Do NOT move LoRA-specific logic into SyncState. SyncState is generic state management. apply_lora_dropout knows about lora_A parameters. Keep them separate.
Do NOT keep any module-level state flags. After this implementation, _dropout_active, _restore_failed, _initialized, _engine_id, _cache_potentially_stale should not exist anywhere except inside SyncState.
Do NOT skip the context manager. Manual enter/exit calls get forgotten. The context manager is the mechanism that makes cleanup impossible to skip.
Drift Patterns

What Would Go Wrong Later

If we kept scattered flags:

Every new feature that touches weight sync would need to understand which flag lives where. New developers would add their own flags. The 5 flags would become 8, then 12. Each flag would have its own lock (or none). Race conditions would become unfixable because no one would understand the full state space.

If we used TrainingContext:

TrainingContext would grow to hold both config AND state. The boundary between "things that configure a run" and "things that track runtime state" would blur. Eventually someone would try to serialize TrainingContext and wonder why it has transient state. Or try to share it across workers and get race conditions.

If we skipped the context manager:

The enter_dropout/exit_dropout pattern would work... until someone adds a new code path and forgets exit_dropout. The "dropout applied twice without restore" warning would become common. Eventually someone would disable the warning instead of fixing the bug.
