Implementation Plan

StateSpec Architecture

Explicit state contracts via typed dataclasses. One source of truth for mutable state. Compile-time enforcement of checkpoint compatibility.

Architecture

New Components

All new StateSpec dataclasses live in qgre/types.py, extending the existing type definitions. Each component holds a single _state slot referencing its StateSpec.

TrainerState

All mutable trainer state in one place.

  • global_step: int
  • accumulated_loss: float
  • accumulation_count: int
  • accumulated_samples: int
  • resumed_mid_accumulation: bool
  • fused_validated: bool
  • needs_weight_sync: bool

DataLoaderState

Epoch tracking, sampling weights, and curriculum gating.

  • epoch: int
  • step_in_epoch: int
  • total_steps: int
  • priority_weights: list[float] | None
  • difficulty_gate: tuple[set, str] | None

AdvantageEstimatorState

Estimator configuration and accumulated statistics.

  • clip_advantage: float
  • step_region_map: dict | None

WeightLoaderState

LoRA initialization flags and cleanup tracking.

  • load_lora_called: bool
  • initialized: bool
  • cleaned_up: bool

CheckpointState

Master container holding all component states. This is what gets serialized.

  • trainer: TrainerState
  • dataloader: DataLoaderState
  • advantage_estimator: AdvantageEstimatorState
  • weight_loader: WeightLoaderState
  • game_state: GameState
  • vprm_critic_state: dict | None
  • vprm_optimizer_state: dict | None
  • schema_version: int
Data Flow

How State Moves Through the System

Principle: State is always accessed via the StateSpec. Components never store mutable fields outside their _state slot. This guarantees that serializing the StateSpec captures ALL mutable state.

Save Path

trainer._state + dataloader._state + ... → CheckpointState → dataclasses.asdict() → torch.save()

Load Path

torch.load() → CheckpointState(**dict) with validation → distribute to components

Validation

On load, CheckpointState.__post_init__() validates:

  • All required fields present (missing → ValueError)
  • All field types correct (mismatch → TypeError)
  • Extra fields logged as warning (forward compatibility)
  • Schema version compatible (migration if needed)

Migration strictness (uncertainty: 0.6)

Specificity

What Surprised Us

The PyTorch Module pattern (state_dict/load_state_dict) isn't actually broken. The issue is that non-Module components (trainer, dataloader, etc.) don't have any enforced contract. StateSpec extends PyTorch's pattern to non-Module components without fighting the framework.

Pattern Memory

Antibody Deposits

Implicit state → Explicit StateSpec

When adding mutable state to a training component, always add it to the component's StateSpec dataclass. Never add loose self.field attributes. The StateSpec IS the checkpoint contract.

Runtime errors → Design-time type errors

Checkpoint bugs should surface as TypeErrors during development (missing field in StateSpec), not as silent divergence in production. Friction belongs at design time.

Distributed ownership → Centralized declaration

Each component owns its behavior, but state declarations are centralized in types.py. This prevents the N-way sync problem between __init__, state_dict, and checkpoint code.
