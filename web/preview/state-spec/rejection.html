Core Rejection

The Implicit State Pattern

Do NOT continue with implicit state management. The current pattern (scatter mutable fields across __init__, manually sync to state_dict, manually sync to checkpoint) is the root cause of 17+ production bugs. No amount of "fixing individual bugs" addresses the structural weakness.

The implicit pattern feels natural because it matches how most Python classes work. But RL training has a unique constraint: state must survive process boundaries. What feels like normal OOP becomes a sync nightmare when checkpointing enters the picture.

Rejected Approaches

Alternatives Considered and Dismissed

Protocol-based Contracts

Define a Stateful Protocol with state_dict() and load_state_dict(). Type checker enforces the contract.

Why rejected: Documents the contract but doesn't enforce field transcription. You still manually keep __init__ and state_dict in sync. The Protocol catches "method missing" but not "field missing".

Immutable State + Functional Updates

All state is immutable. Updates return new state objects. Checkpoint is trivially the current state object. Elm/Redux style.

Why rejected: Would require rewriting the entire trainer to use functional updates. The insight (immutability simplifies state) is sound, but the migration cost is prohibitive for this codebase.

Descriptor-based Auto-tracking

Use Python descriptors to automatically track mutable attributes. Checkpoint includes all descriptor-tracked fields.

Why rejected: Magic. Fragile. Hard to debug. When the descriptor fails to track a field, you get the same bug class we're trying to eliminate.

Just Add More Tests

Continue with implicit state but add linting/tests that detect state/checkpoint drift.

Why rejected: Tests detect drift after it happens. The 17 findings we fixed were all cases where tests weren't catching drift. Prevention beats detection.

Assumptions Overturned

What Training Suggested vs What Evidence Shows

Assumption Evidence
"Each component should manage its own state" Distributed ownership causes sync failures. Centralize state declarations.
"state_dict/load_state_dict is the established pattern" PyTorch pattern works for Modules. Non-Module components need explicit StateSpec.
"Runtime errors are acceptable for checkpoint issues" Runtime errors cause production failures. Move friction to design time via type errors.
"We can fix bugs as they arise" We fixed 59 bugs in 3 rounds. The pattern keeps producing bugs. Fix the pattern.
Drift Patterns

Bug Classes That Will Recur Without StateSpec

State Field Not Saved

Developer adds self.new_field to __init__. Forgets to add to state_dict. Checkpoint doesn't include it. Resume loses the value. Training diverges silently.

State Field Not Restored

Field is in checkpoint but load_state_dict doesn't read it. Old value persists. Metrics show wrong values. Debugging takes hours.

Default Mismatch

Field has default in __init__ (e.g., self.count = 0). Checkpoint has count = 5. Resume calls __init__ first, sets count=0, then load_state_dict overwrites. But if load_state_dict uses .get() with wrong default, you get the wrong value.

Flag Not Reset

Boolean flag like _resumed_mid_accumulation is set on resume. Never cleared. Next step still thinks it's resuming. Momentum corruption.
